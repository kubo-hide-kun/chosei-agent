# プルリクエスト指示書

## 目的

Gitmoji を使ったプルリクエスト（PR）を日本語で統一し、レビュー時に「この PR で何を成し得るのか」「なぜ必要なのか」「どこを確認すべきか」を素早く判断できる状態にする。

## 最重要ルール

1. **PR タイトル・本文は日本語で書く。**
2. **PR タイトルは変更内容ではなく、アウトカムを簡潔に表現する。**
3. **PR タイトル形式は `type(scope): outcome :gitmoji:` に統一する。**
4. **Gitmoji は末尾に shortcode で書く。**
5. PR 本文には「目的」「背景」「成し得たこと」「確認内容」「リスク」を書く。
6. レビュー依頼前に、WIP コミット、不要なデバッグログ、不要なコメントアウトを整理する。

## PR タイトル

### 基本フォーマット

```text
<type>(<scope>): <アウトカム> <gitmoji-shortcode>
```

例:

```text
feat(auth): パスワードを忘れたユーザーが自力で再設定できるようにする :sparkles:
fix(billing): 請求金額が正しく表示されるようにする :bug:
refactor(user): ユーザー処理の責務を追いやすくする :recycle:
```

`scope` が不要な場合のみ、次の形式を許容する。

```text
<type>: <アウトカム> <gitmoji-shortcode>
```

例:

```text
docs: 初期セットアップを迷わず実行できるようにする :memo:
```

### タイトル例

| 悪い例: 変更内容 | 良い例: アウトカム |
|---|---|
| `feat(auth): パスワード再設定画面を追加 :sparkles:` | `feat(auth): パスワードを忘れたユーザーが自力で再設定できるようにする :sparkles:` |
| `fix(billing): 請求金額の計算式を修正 :bug:` | `fix(billing): 請求金額が正しく表示されるようにする :bug:` |
| `refactor(user): ユーザーサービスを分割 :recycle:` | `refactor(user): ユーザー処理の責務を追いやすくする :recycle:` |
| `test(auth): ログインテストを追加 :white_check_mark:` | `test(auth): ログイン失敗時の挙動を継続的に検証できるようにする :white_check_mark:` |
| `chore(session): Cookie 設定を変更 :lock:` | `fix(session): セッション Cookie の漏えいリスクを下げる :lock:` |

## type の使い分け

| type | 用途 | PR タイトル例 |
|---|---|---|
| `feat` | 新機能・新しい価値の追加 | `feat(auth): パスワードを忘れたユーザーが自力で再設定できるようにする :sparkles:` |
| `fix` | 不具合修正 | `fix(billing): 請求金額が正しく表示されるようにする :bug:` |
| `hotfix` | 本番影響の緊急修正 | `hotfix(api): 本番環境でのクラッシュを防ぐ :ambulance:` |
| `refactor` | 振る舞いを変えない内部改善 | `refactor(user): ユーザー処理の責務を追いやすくする :recycle:` |
| `perf` | パフォーマンス改善 | `perf(search): 検索結果の取得時間を短縮する :zap:` |
| `style` | UI・見た目・フォーマット | `style(button): 主要アクションを視認しやすくする :lipstick:` |
| `test` | テスト追加・更新 | `test(auth): ログイン失敗時の挙動を継続的に検証できるようにする :white_check_mark:` |
| `docs` | ドキュメント追加・更新 | `docs(readme): 初期セットアップを迷わず実行できるようにする :memo:` |
| `build` | ビルド・依存関係 | `build(deps): 入力値のスキーマ検証を利用できるようにする :heavy_plus_sign:` |
| `ci` | CI/CD | `ci(actions): PR ごとにテストを自動実行できるようにする :construction_worker:` |
| `chore` | 設定・雑務 | `chore(eslint): コーディング規約違反を検知できるようにする :wrench:` |
| `revert` | 変更取り消し | `revert(auth): パスワード再設定の公開を取り消してログイン影響を回避する :rewind:` |

- [ ] Integration tests

## 各項目の書き方

### 目的

この PR の最重要項目。変更内容ではなく、実現される状態を書く。

良い例:

```md
パスワードを忘れたユーザーが、問い合わせをせずに自力でアカウントへ復帰できるようにする。
```

悪い例:

```md
パスワード再設定画面を追加する。
```

### 背景

判断の文脈を書く。仕様、障害、問い合わせ、運用負荷、技術的負債など、なぜ今必要かを明確にする。

### 成し得たこと

レビュー前に、PR の成果をアウトカムとして列挙する。

例:

```md
- ユーザーがログインできない状態から自力で復帰できる
- CS へのパスワード再設定依頼を減らせる
- 再設定メールの送信失敗を検知できる
```

### 主な変更点

ここでは実装上の変更内容を書いてよい。ただし、目的やタイトルの代わりにしない。

例:

```md
- パスワード再設定申請 API を追加
- 再設定メール送信処理を追加
- 再設定トークンの有効期限チェックを追加
```

### レビュー観点

レビュアーが見るべき観点を明示する。

例:

```md
- トークンの有効期限と再利用不可の扱いが妥当か
- エラーメッセージからアカウント存在有無が推測できないか
- 既存ログイン導線への副作用がないか
```

## PR 作成前チェックリスト

- [ ] PR タイトルが `type(scope): outcome :gitmoji:` 形式になっている
- [ ] Gitmoji が末尾に shortcode で書かれている
- [ ] PR タイトルが日本語で書かれている
- [ ] PR タイトルが変更内容ではなくアウトカムになっている
- [ ] 目的に「何を成し得るのか」が書かれている
- [ ] 背景に「なぜ必要か」が書かれている
- [ ] 成し得たことが変更内容の羅列になっていない
- [ ] 主な変更点がレビュー補助として整理されている
- [ ] レビュー観点が具体的に書かれている
- [ ] 確認内容が書かれている
- [ ] UI 変更がある場合、スクリーンショットまたは動画がある
- [ ] DB / 環境変数 / Feature Flag / デプロイ順序の影響を書いた
- [ ] 破壊的変更がある場合、`:boom:` と `BREAKING CHANGE` を明記した
- [ ] 不要な WIP コミット・デバッグログ・コメントアウトを残していない

## squash merge する場合

- PR タイトルを最終コミットメッセージとして扱う。
- PR タイトルは `type(scope): outcome :gitmoji:` 形式にする。
- Gitmoji は末尾に shortcode で書く。
- 複数の意図がある場合は、最もユーザー影響またはリリース影響が大きいものを選ぶ。
- タイトルは日本語のアウトカム表現にする。

例:

```text
feat(auth): パスワードを忘れたユーザーが自力で再設定できるようにする :sparkles:
```

## merge commit する場合

- 各コミットが独立して読める状態にする。
- `fix`, `update`, `対応`, `修正`, `WIP` などの曖昧なコミットは merge 前に整理する。
- 各コミットも `type(scope): outcome :gitmoji:` 形式にする。
- 各コミットも日本語のアウトカム表現にする。

## revert する場合

```text
revert(<scope>): <取り消しによって実現する状態> :rewind:
```

例:

```text
revert(auth): パスワード再設定の公開を取り消してログイン影響を回避する :rewind:
```

本文には、revert 理由と再適用条件を書く。

## WIP の扱い

- 個人作業中の一時 PR には `:construction:` を使ってよい。
- レビュー依頼前に、原則として WIP 状態を解除する。
- 意図的に WIP のままレビュー依頼する場合は、PR タイトルまたは目的に `WIP` と理由を書く。

例:

```text
chore(dashboard): 指標画面の土台を確認できるようにする :construction:
```
