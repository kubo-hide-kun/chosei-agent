# ADR 0002: 入稿 JSON フォーマットの設計

- ステータス: Accepted
- 日付: 2026-07-11

## 背景

既存の日程調整サービスの課題は、候補日時をボタン操作で 1 件ずつ登録する入稿コスト。
本サービスの中核価値は「候補日時をまとめて入稿できる」ことなので、その形式が UX を決める。

## 決定

`{ title, description?, candidates[] }` を入稿 JSON とし、candidates は **2 形式の混在を許可**する。

1. オブジェクト形式: `{ "date": "YYYY-MM-DD", "start"?: "HH:mm", "end"?: "HH:mm", "label"?: string }`
2. 文字列ショートハンド: `"YYYY-MM-DD"` / `"YYYY-MM-DD HH:mm"` / `"YYYY-MM-DD HH:mm-HH:mm"`

- 仕様の正本は [docs/JSON_FORMAT.md](../JSON_FORMAT.md)。スキーマ実装は `src/server/domain/event.ts`
- バリデーションエラーは 422 で `{ error, issues: [{ path, message }] }` を返す(メッセージは日本語)
- `label` 省略時は `7/21(火) 19:00〜21:00` 形式で自動生成する

## 理由

- 手書き・スクリプト生成・AI 生成のどの経路でも書きやすいこと(ショートハンドは手書き向け、オブジェクトはプログラム生成向け)
- フィールド単位のエラー(`issues[].path`)を返すことで、大きな JSON を貼ったときも修正箇所がすぐ分かる

## 影響・トレードオフ

- 2 形式の混在を許すぶん、正規化(`normalizeCandidate`)の分岐とテストが増える。UX 優先で許容
- 日付は `YYYY-MM-DD` 固定(タイムゾーンは扱わない)。国際化が必要になったら別 ADR で判断

## 代替案

- **単一のオブジェクト形式のみ**: バリデーションは単純になるが、手書き入稿の負担が増えるため不採用
- **CSV / スプレッドシート貼り付け**: 需要が確認できたら追加検討(JSON と排他ではない)
