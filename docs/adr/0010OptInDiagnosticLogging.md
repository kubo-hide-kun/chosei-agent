# ADR 0010: 同意ベースの診断ログ(Claude JSON parse 失敗時のみ)

- ステータス: Accepted
- 日付: 2026-07-13

## 背景

ADR 0008/0009 は「PII(名前・コメント・自然文入力の本文)はログに残さない」を不変条件としている。
そのため `agent.claude_fallback`(ADR 0009)には内容を含まない固定メッセージと非 PII な診断値
(`rawLength` / `stopReason` / `hasJsonStart` / `truncated` / `errorPosition`)しか残せず、
「Claude が出力の途中で説明文を付けた」「特定の入力パターンで毎回壊れる」といった
**内容に踏み込んだ原因調査**ができなかった。

本サービスにはユーザーアカウントが無く(合言葉ベースの共有 URL モデル、ADR 0006)、
事前に「ログを残してよい」という設定を保存できる場所も無い。そのため同意は
**入力フォームごとにその都度取る**以外の選択肢が無い。

## 決定

1. AI 解析を呼び出す 2 つの入力フォーム(イベント作成の自然文入力、出欠回答の自然文入力)に
   任意のチェックボックス「AI の解析に失敗した場合、原因調査のため入力内容と AI の応答を
   運営のログに一時的に記録することに同意する」を追加する(既定は未チェック)
2. チェックボックスの状態は API リクエストボディの `allowDiagnosticLogging: boolean` として
   送信し、`parseScheduleText` / `parseAnswerText` に引き渡す
3. `allowDiagnosticLogging` が true のときに限り、かつ **解析が実際に失敗したときのみ**、
   `agent.claude_fallback` ログに次を追加する:
   - `inputText`: ユーザーが入力した自然文そのもの
   - `rawResponse`: Claude の生応答テキスト(`ClaudeJsonParseError.rawResponse`)
   - `diagnosticConsent: true`(このログ行に同意ベースの内容が含まれることを示す目印)
4. 同意が無い場合(既定)は従来どおり ADR 0009 の非 PII フィールドのみを記録する
5. ログの保存先は既存の構造化ログ(stdout → Railway Logs)をそのまま使う。新しい保存先は作らない

## 理由

- フォーム単位の都度同意は、アカウントが無いこの構成で唯一実現可能な同意の粒度であり、
  「その回だけ、その入力について」同意する UX は取得元の入力を対象にした説明として自然
- `ClaudeJsonParseError` は `rawResponse` を常時保持するが、ログに載せるかどうかは
  use case 層(`parseScheduleText` / `parseAnswerText`)が同意フラグを見て判断する。
  parse 層(`claudeJson.ts`)は PII 判断をしない(責務を分離する)
- 解析が成功した場合は診断情報自体が発生しない(ログに出るのは失敗時のみ)ため、
  同意してもらう頻度・影響範囲は最小限になる

## 影響・トレードオフ

- 同意した場合、Railway Logs にユーザーの自然文入力と AI 応答が平文で残る。
  保持期間・アクセス制御は Railway のログ基盤に依存する(ADR 0008 と同じ制約)
- チェックボックスの文言は「原因調査のため」「運営のログに一時的に記録」という
  リスクの要点のみを伝える簡潔な説明にした。詳細な法的文言(プライバシーポリシー等)は
  本サービスの規模ではスコープ外
- 同意は API リクエストごとに検証されるが、フォームの入力欄自体が信頼された同意主体という
  前提(合言葉モデルと同じく「URL / フォームにアクセスできる人」を信頼する設計)

## 代替案

- **運営全体で一律 ON/OFF(環境変数)**: ユーザー個別の同意にならないため不採用
- **イベント単位で保存**: 参加者の回答に幹事の設定が無断で適用される(参加者は同意していない)
  ため不採用。フォーム単位(その場で入力した本人の同意)を採用
- **別の保存先(専用テーブル等)を新設**: 現状の規模では過剰。要件が増えたら別 ADR で判断
