# ADR 0013: 自然文によるイベント編集(AI 編集エージェント)

- ステータス: Accepted
- 日付: 2026-07-13

## 背景

[ADR 0011](./0011EventEditing.md) でイベント編集(title・description・candidates)は可能になったが、
編集手段は JSON の直接編集のみだった。作成側([ADR 0003](./0003AgentParsingStrategy.md))・
回答側([ADR 0005](./0005AiAnswerInput.md))は自然文入力に対応しているのに対し、編集だけ JSON 必須なのは
一貫性がなく、候補の追加や日程変更のたびに JSON を手で書き換える負担があった。

## 決定

1. **3 つ目の AI エージェントを追加する**: 「現在の入稿 JSON + 自然文の変更指示」→「変更を反映した新しい入稿 JSON」
   を出力するイベント編集エージェントを追加する。プロンプトの正本は
   [docs/SYSTEM_PROMPT.md](../SYSTEM_PROMPT.md) §3(`gateways/eventEditPrompt.ts` と同期必須)
2. **フォールバックは持たない**: 候補日時抽出・出欠回答抽出は Claude + ルールベースの 2 段構え(ADR 0003)だが、
   「任意の自然文の編集指示」を規則で再現するのは現実的でないため、この機能は Claude 専用とする。
   API キー未設定・予算超過・解析失敗時は「JSON を直接編集してください」という案内エラーを返し、
   既存の JSON 直接編集(ADR 0011)に確実に降りられるようにする
3. **human in the loop を維持する**: ADR 0005 と同じく、解析結果は直接保存せず、編集画面の JSON テキストエリアに
   反映するだけに留める。利用者が内容を確認し、既存の `PUT /api/events/{id}`(ADR 0011)で保存する
4. **アクセス制御**: 編集そのもの(`PUT`)は合言葉不要(ADR 0011)だが、この機能は Claude API を呼びコストが発生するため、
   作成側の AI 解析・回答側の AI 解析と同じく合言葉必須・レートリミット・日次予算の対象とする(ADR 0006)
5. **実装**: `gateways/claudeEventEditGateway.ts`(Claude 呼び出し)→ `useCases/parseEventEditText.ts`(呼び出し制御・
   ログ)→ `POST /api/events/[id]/agent/parse-edit`(認可・レートリミット・予算消費)。
   基準日文字列の組み立ては候補日時抽出エージェントと重複していたため `gateways/baseDate.ts` に共通化した

## 理由

- 編集の合言葉ポリシーを「Claude を呼ぶかどうか」で分けるのは ADR 0006 の既存方針(「合言葉は作成・AI 解析のみ」)の
  自然な延長であり、新しい認可モデルを増やさずに済む
- 候補の同一性(date/start/end)による差分適用は ADR 0011 がすでに解決済みのため、
  この機能は「新しい入稿 JSON を組み立てる」ことだけに専念すればよく、保存処理を再実装しない

## 影響・トレードオフ

- AI 解析エンドポイントが 1 つ増え、悪用時のコスト面がさらに拡大する → ADR 0006 の対策を同時に適用
- 変更しないつもりの候補でも AI が date/start/end を書き換えてしまうと、意図せず既存回答を失う可能性がある。
  プロンプトで明示的に禁止しているが、解析結果は保存前にプレビューで確認できるため最終防御はユーザー確認に委ねる
- ルールベースフォールバックがないため、Claude 未設定/予算超過の環境ではこの機能を使えない
  (JSON 直接編集は常に利用できるため、機能が完全に失われるわけではない)

## 代替案

- **既存の候補日時抽出エージェントを流用し、既存 JSON をプロンプトに含めない**: 既存候補の温存(date/start/end を
  変えない)という制約を表現できず、編集のたびに全候補が作り直されて回答が失われるため不採用
- **編集結果を即保存**: 誤読時に取り消せない(ADR 0005 と同じ懸念)ため不採用。プレビュー反映方式とした
