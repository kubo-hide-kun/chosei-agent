# ADR 0011: イベント編集機能の追加

- ステータス: Accepted
- 日付: 2026-07-13

## 背景

これまでイベントは作成のみ可能で、タイトル・説明・候補日時を後から直す手段がなかった
(誤字や日程漏れがあると、イベントを作り直して URL を共有し直すしかなかった)。
利用者からの指摘を受け、編集導線を追加する。

## 決定

1. **編集範囲**: title・description・candidates をすべて編集可能にする。入稿 JSON([ADR 0002](./0002JsonImportFormat.md))と同一スキーマを使い回す
2. **アクセス制御**: イベント閲覧・出欠回答([ADR 0006](./0006AbuseProtection.md))と同じモデルを踏襲し、
   イベント URL(`id`)を知っている人は誰でも編集できる。合言葉(アクセスキー)は要求しない
   (編集は Claude API を呼ばずコスト面の懸念がないため、`ADR 0006` の「合言葉は作成・AI 解析のみ」の対象外)
3. **候補日時の差分適用**: 新しい candidates は `(date, start, end)` で既存候補と照合する。
   一致する候補は同じ内部 ID を引き継ぎ、紐づく既存回答を保持する。
   一致しない(=リストから消えた)候補は削除し、その候補への回答は cascade delete で破棄する。
   一致しない新しい候補は新規候補として追加する
4. **実装**: `EventRepository.updateEvent` を追加し、`SqliteEventRepository` で 1 トランザクション内に
   `UPDATE events` + 候補の delete/upsert を行う。API は `PUT /api/events/[id]`(既存の `GET` と同じルート)

## 理由

- 認証基盤を持たないアプリで編集だけ別の認可モデルにすると一貫性がなく、利用者にも分かりにくい。
  「URL を知っている = 幹事から共有された身内」という既存の信頼モデルをそのまま延長するのが最小の変更
- 候補日時を ID ではなく内容で照合するのは、入稿 JSON が候補 ID を含まない(サーバー生成のため)ため。
  「日付・時刻が同じなら同じ候補」という直感的な同一性の定義は、UI 側でも実装コストが低い

## 影響・トレードオフ

- 誰でも編集できるため、URL を共有された第三者が悪意を持てば内容を改ざんできる
  (既存の「誰でも回答できる」と同じリスクレベル。悪用時は幹事が合言葉付きの新イベントに切り替える運用で対応)
- 同じ `(date, start, end)` で `label` だけ変えても回答は保持される一方、時刻を 1 分でもずらすと
  別候補扱いになり回答が失われる。候補の「同一性」が暗黙知になる点は [docs/USER_GUIDE.md](../USER_GUIDE.md) で明示する
- 同時編集の排他制御はない(最後に保存した内容で上書きされる)。低頻度・少人数利用が前提のため許容

## 代替案

- **候補 ID を入稿 JSON に含めて明示的に指定させる**: クライアント側の実装・入稿 JSON の複雑化に見合わないため不採用
- **候補日時の変更を禁止し title・description のみ編集可**: 「候補を追加したい」という主要な要望に応えられないため不採用
- **編集用の別トークン(幹事のみ)を新設**: 認可の概念を新規導入するコストが高く、現状のリスク許容度に対して過剰なため不採用(将来、不正利用が問題化したら再検討)
