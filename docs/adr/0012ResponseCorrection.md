# ADR 0012: 出欠回答の修正(同名再送信は上書き)

- ステータス: Accepted
- 日付: 2026-07-13

## 背景

出欠回答(◯/△/✕・名前・コメント)を後から修正する手段がなかった([Issue #6](https://github.com/kubo-hide-kun/chosei-agent/issues/6))。
`addResponse` は常に新規 `responses` 行を insert するため、同じ名前で再送信すると集計表に別行として重複表示され、
参加者は自分では直せず幹事への連絡が必要だった。

## 決定

1. **本人確認の方法**: [ADR 0011](./0011EventEditing.md) と同じ「URL を知っている人は誰でも」という信頼モデルを踏襲し、
   **同じ名前(前後の空白を除去して完全一致)で再送信した人は誰でも上書きできる**。ログインや編集トークンは導入しない
2. **上書き挙動**: `(event_id, name)` が一致する既存回答があれば、新規 insert ではなく**その回答を更新**する
   (コメント・回答内容を丸ごと置き換え)。一致しなければ新規回答として追加する。
   `created_at` は変更しないため、集計表での列の並び順は変わらない
3. **名前の正規化**: 入力名は前後の空白を trim してから比較・保存する(`responseSchema` 側で trim)。
   大文字小文字や全角半角の揺れは正規化しない(過剰実装を避ける。同一人物が表記を変えて送信した場合は別行のままになる)
4. **UI**: 名前欄に既存回答者と一致する名前を入力すると、「既存の回答を上書きします」旨を表示し、
   既存の回答内容(◯/△/✕・コメント)を読み込んで編集できるボタンを設ける(ゼロから入力し直す必要をなくす)
5. **重複行(既知のバグ的挙動)の扱い**: 本 ADR 以降の新規回答から上書きが有効になる。
   本 ADR 以前に生成済みの重複行を自動マージする移行処理は行わない(低頻度利用が前提でリスクが小さいため)

## 理由

- 新しい認可の概念(編集トークン等)を導入するコストは、[ADR 0011](./0011EventEditing.md) で「編集用の別トークン」を
  不採用とした理由と同じくリスク許容度に対して過剰。既存の「名前」を鍵にする方が実装・UX ともに最小
- DB マイグレーション機構が未導入([docs/steering/repo-map.md](../steering/repo-map.md))のため、
  `UNIQUE` 制約をスキーマに追加せず、`addResponse` 内で「既存行を検索 → あれば UPDATE、なければ INSERT」という
  アプリケーション側の判定にする。既存 DB に対しても再起動だけで動作する

## 影響・トレードオフ

- 同じ名前を騙って第三者が他人の回答を書き換えられる(既存の「誰でも回答できる」「誰でもイベントを編集できる」と
  同じリスクレベル。悪用時は幹事が合言葉付きの新イベントに切り替える運用で対応)
- 家族・同姓同名など、複数人が本当に同じ名前で回答したい場合は上書きし合ってしまう
  (回避策として名前に識別子を付けてもらう運用を [docs/USER_GUIDE.md](../USER_GUIDE.md) に明記する)
- 表記揺れ(スペース以外の空白・大文字小文字・全角半角)は正規化しないため、揺れがあると別回答として扱われる

## 代替案

- **回答直後に発行する編集用リンク(トークン)を都度発行する**: 認可の概念を新規導入するコストが高く、
  リンクの保管・共有という新たな UX 負担も生む。現状のリスク許容度に対して過剰なため不採用
  (将来、なりすましによる悪用が問題化したら再検討)
- **明示的な「回答を編集する」導線(既存回答一覧からの選択)を別途設ける**: 同名判定に加えて追加の UI 状態管理が必要になり、
  「同じ名前で送信すれば直せる」という直感的な挙動だけで要件を満たせるため不採用。
  ただし UI 上で上書き対象を可視化する最小限の導線(決定 4)は設ける
