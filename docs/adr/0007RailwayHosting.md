# ADR 0007: ホスティングを Railway(単一インスタンス + 永続ボリューム)にする

- ステータス: Accepted
- 日付: 2026-07-11

## 背景

本システムには、ホスティング先を決める 2 つの制約がある(いずれも意図的な設計判断)。

1. SQLite がローカルディスクに書く(ADR 0001)→ 永続ボリューム付きの常駐プロセスが必要
2. レートリミット・Claude 日次予算がインメモリ(ADR 0006)→ 単一プロセス前提

身内利用の低トラフィックが前提なので、「小さいサーバー 1 台の立てっぱなし」が最も要件に合う。

## 決定

- **Railway に GitHub 連携でデプロイする**。ビルドはリポジトリの `Dockerfile`、
  デプロイ挙動(ヘルスチェック・リスタート・レプリカ数 1)は `railway.json` を正本とする
- 永続ボリュームを `/data` にマウントし、`CHOSEI_DATA_DIR=/data`(Dockerfile で設定済み)で SQLite を置く
- **インスタンスは 1 台に固定**(`railway.json` の `numReplicas: 1`)
- 運用手順の正本は [docs/runbook/railway-deploy.md](../runbook/railway-deploy.md)

## 理由

- コード変更ゼロで現行構成のまま動く(サーバーレス系は DB 移行・レートリミット外部化が必要)
- GUI 完結(ボリューム・環境変数・ドメイン)で運用が最も軽く、GitHub push だけでデプロイできる
- 従量課金で身内利用の負荷なら最低額に収まる

## 影響・トレードオフ

- 東京リージョンがない(最寄りはシンガポール)。日程調整の用途ではレイテンシ影響は小さいと判断
- スケールアウト不可(仕様として固定)。負荷が問題になったら DB 移行 + レートリミット永続化を先に ADR で判断する
- Dockerfile のビルドステージに python3/make/g++ を含める(better-sqlite3 のプレビルドが無い
  Node バージョンでもソースビルドで通すため。ローカル検証でプレビルド欠落による失敗を確認済み)

## 代替案

- **Render**: 同等の構成が可能だがディスク単価が高く固定費も上。今回の要件での優位なし
- **Fly.io**: 東京リージェンあり・常駐最安だが CLI 前提でセットアップの手間が一段多い。レイテンシ要件が出たら再検討
- **Vercel / Cloudflare**: DB・レートリミットの外部化が必要で「楽さ」の要件に反する。スケールが必要になった段階の選択肢
